# Pick Deal 요구사항 초안

## 1. 프로젝트 목적

Pick Deal은 여러 커뮤니티와 쇼핑 관련 사이트에 흩어진 핫딜 정보를 개인 기준으로 모아 보는 개인용 핫딜 큐레이션 서비스이다.

초기 목표는 실제 크롤링 자동화나 AI 요약보다, 사용자가 원하는 출처와 키워드 기준으로 핫딜을 탐색할 수 있는 기본 제품 구조를 먼저 검증하는 것이다.

핵심 가치는 다음과 같다.

- 여러 출처의 핫딜 정보를 하나의 목록에서 확인한다.
- 관심 있는 출처만 표시하고, 원하지 않는 출처는 숨긴다.
- 관심 키워드와 제외 키워드를 이용해 개인화된 탐색 기준을 만든다.
- 이후 자동 수집기, 가격 추적, 댓글 요약, 알림 기능으로 확장할 수 있는 기반을 만든다.

## 2. MVP 기능 범위

### 포함 기능

- 핫딜 목록 조회
- 핫딜 상세 조회
- 출처 사이트 목록 조회
- 출처 사이트 표시/숨김 설정
- 관심 키워드 등록, 조회, 삭제
- 제외 키워드 등록, 조회, 삭제
- 더미 데이터 또는 수동 등록 데이터 기반 운영

### 제외 기능

- 실제 커뮤니티/쇼핑몰 크롤링
- 로그인/회원가입
- AI 요약
- 댓글 수집
- 알림 발송
- 결제/수익화
- 관리자 화면
- 브라우저 확장 프로그램

## 3. 사용자 시나리오

### 핫딜 탐색

1. 사용자는 핫딜 목록 화면에 진입한다.
2. 백엔드는 표시 상태가 활성화된 출처의 핫딜을 반환한다.
3. 프론트엔드는 제목, 가격, 출처, 등록 시간, 원문 링크를 목록에 표시한다.
4. 사용자는 특정 핫딜을 클릭해 상세 화면으로 이동한다.

### 출처 설정

1. 사용자는 출처 설정 화면에 진입한다.
2. 전체 출처 사이트 목록과 현재 표시 여부를 확인한다.
3. 사용자는 출처별 표시/숨김 상태를 변경한다.
4. 변경된 설정은 이후 핫딜 목록 조회에 반영된다.

### 키워드 설정

1. 사용자는 키워드 설정 화면에 진입한다.
2. 관심 키워드와 제외 키워드 목록을 확인한다.
3. 사용자는 키워드를 추가하거나 삭제한다.
4. 백엔드는 키워드 설정을 저장하고 목록 필터링 기준으로 사용할 수 있게 한다.

## 4. MVP 정책

### 데이터 정책

- 초기 데이터는 DB seed, 수동 insert, 또는 개발용 fixture로 구성한다.
- 핫딜 원문은 외부 URL로 연결하고, 본문 전체를 복제 저장하지 않는다.
- 가격, 배송비, 품절 여부 등은 가능한 경우 별도 필드로 분리하되 MVP에서는 nullable로 둔다.

### 필터링 정책

- 표시 상태가 `visible = true`인 출처만 기본 목록에 노출한다.
- 관심 키워드가 존재하면 제목 또는 설명에 해당 키워드가 포함된 핫딜을 우선 노출하거나 필터링할 수 있다.
- 제외 키워드가 포함된 핫딜은 목록에서 제외한다.
- MVP에서는 복잡한 검색 엔진을 도입하지 않고 DB `LIKE` 또는 단순 문자열 매칭으로 시작한다.

### 사용자 범위

- MVP는 개인용 단일 사용자 서비스를 기준으로 한다.
- 별도 사용자 계정 테이블 없이 전역 설정으로 출처/키워드 설정을 저장한다.
- 추후 다중 사용자 지원 시 `users` 테이블과 사용자별 설정 테이블로 확장한다.

## 5. 비기능 요구사항

- 프론트엔드는 Next.js App Router와 TypeScript를 사용한다.
- 스타일링은 Tailwind CSS를 사용한다.
- 백엔드는 Spring Boot 기반 REST API로 구현한다.
- DB 연동을 전제로 설계한다.
- API 응답은 프론트엔드에서 다루기 쉬운 JSON 형태로 제공한다.
- MVP 단계에서는 배치/수집/AI 처리와 API 서버를 분리하지 않아도 되지만, 구조상 분리 가능하게 설계한다.

## 6. 성공 기준

- 사용자가 더미/수동 등록 핫딜 목록을 볼 수 있다.
- 사용자가 핫딜 상세 정보를 볼 수 있다.
- 사용자가 출처 목록을 확인하고 표시 여부를 바꿀 수 있다.
- 사용자가 관심 키워드와 제외 키워드를 관리할 수 있다.
- 설정 변경이 목록 조회 결과에 반영될 수 있는 API/DB 구조가 준비된다.
