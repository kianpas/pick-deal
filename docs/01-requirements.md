# 01. 요구사항 정의서 (Requirements)

> PickDeal — 개인용 핫딜 큐레이션 서비스
> 본 문서는 이후 구현 작업의 기준 문서입니다. 모든 설계 문서(`docs/01` ~ `docs/06`)는 상호 참조됩니다.
> 작성 기준일: 2026-05-20

---

## 1. 프로젝트 목적

PickDeal은 여러 커뮤니티/사이트에 흩어져 있는 핫딜 정보를 한곳에 모아, **사용자가 원하는 출처와 키워드 기준으로만 걸러서 보는** 개인용 큐레이션 서비스다.

핵심 가치는 다음과 같다.

- **집중**: 관심 없는 출처/상품을 숨기고, 관심 키워드 중심으로만 본다.
- **소음 제거**: 제외 키워드로 광고성·관심 밖 딜을 걸러낸다.
- **확장 가능성**: 초기에는 단순 조회/필터에서 시작하되, 수집 자동화·중복 제거·댓글 요약·알림·AI 구매 판단 보조까지 단계적으로 확장한다.

본 프로젝트는 **단일 사용자(개인)** 사용을 전제로 시작한다. 인증/회원 개념은 MVP에 두지 않되, 데이터 모델에는 향후 멀티유저 확장을 위한 여지를 남겨둔다(아래 5장 참고).

---

## 2. 해결하려는 문제

| 문제 | 현재 상황 | PickDeal의 접근 |
| --- | --- | --- |
| 정보 분산 | 핫딜이 여러 커뮤니티/사이트에 흩어져 있음 | 여러 출처의 딜을 한 화면에 모아 조회 |
| 노이즈 | 관심 없는 카테고리·출처·광고성 딜이 섞임 | 출처 표시/숨김 + 관심/제외 키워드 필터 |
| 중복 | 같은 딜이 여러 출처에 중복 게시됨 | (2차) 중복 딜 제거. 단, 초기 스키마에 dedup 키를 미리 반영 |
| 판단 비용 | "이거 살 만한가?"를 매번 직접 판단 | (3차) 댓글 반응 요약 + AI 구매 판단 보조 |
| 놓침 | 관심 딜을 실시간으로 못 봄 | (2차) 관심 키워드 알림 |

MVP 단계에서 실제로 푸는 문제는 **"정보 분산"과 "노이즈"** 두 가지에 한정한다. 나머지는 확장 단계에서 다룬다.

## 3. MVP 기능 범위

> 초기 MVP에서는 **실제 크롤링, AI 요약, 알림 기능을 구현하지 않는다.**
> 더미 데이터 또는 수동 등록 데이터를 기반으로 아래 기능만 먼저 구현한다.

### 3.1 기능 목록

1. **핫딜 목록 조회**
   - 등록된 딜 목록을 페이지네이션으로 조회한다.
   - 출처 필터, 관심/제외 키워드 필터, 정렬(최신순/할인율순)을 지원한다.
   - 사용자가 "숨김" 처리한 출처의 딜은 기본적으로 목록에서 제외한다.
2. **핫딜 상세 조회**
   - 단일 딜의 상세 정보를 조회한다(제목, 가격/정가/할인율, 카테고리, 썸네일, 원본 링크, 출처, 게시 시각 등).
3. **출처 사이트 목록 조회**
   - 등록된 출처 사이트 목록과 각 출처의 표시/숨김 상태를 조회한다.
4. **출처 사이트 표시/숨김 설정**
   - 출처별로 표시 여부를 토글한다. 설정은 백엔드 DB에 저장한다.
5. **관심 키워드 / 제외 키워드 등록·조회·삭제**
   - 관심 키워드(interest)와 제외 키워드(exclude)를 각각 등록, 조회, 삭제한다.
   - 설정은 백엔드 DB에 저장한다.

### 3.2 키워드 필터 동작 규칙 (MVP)

- 필터는 **서버 측에서** 적용한다(클라이언트 필터 아님). 목록 API가 최종 필터링된 결과를 반환한다.
- 매칭 방식은 MVP에서 **단순 포함(substring, 대소문자 무시) 검색**으로 한다. 형태소 분석/토큰화는 도입하지 않는다.
- 적용 우선순위: **제외 키워드 > 관심 키워드 > 출처 숨김**.
  - 제목/요약에 제외 키워드가 하나라도 포함되면 노출하지 않는다.
  - 관심 키워드가 1개 이상 등록되어 있으면, 관심 키워드를 포함하는 딜만 노출한다(관심 키워드가 0개면 출처/제외 기준만 적용).
  - 숨김 처리된 출처의 딜은 노출하지 않는다.
- 위 규칙은 `docs/03-api-design.md`의 목록 조회 API 동작과 일치해야 한다.

### 3.3 데이터 입력 방식 (MVP)

- 실제 크롤링은 구현하지 않는다.
- 더미/시드 데이터(SQL seed 또는 JSON import)로 딜·출처를 채운다.
- 필요 시 운영자가 수동으로 딜을 등록할 수 있는 **내부용 등록 API**를 둘 수 있다(관리 화면은 MVP 범위 밖, API만 선택적으로 제공). 상세는 `docs/03`, 시드 전략은 `docs/04` 참고.

### 3.4 MVP에서 명시적으로 제외하는 것

- 실제 크롤링/수집 자동화
- AI 댓글 요약, AI 구매 판단 보조
- 알림(푸시/이메일 등)
- 중복 딜 제거 로직(스키마만 대비, 로직은 미구현)
- 사용자 인증/회원가입/로그인
- Redis 사용
- Dockerfile / docker-compose.yml / nginx / CI·CD 스크립트 작성

## 4. 확장 기능 범위 (2차 / 3차)

> 확장 기능은 MVP와 **명확히 분리**한다. 아래 항목은 MVP 구현 대상이 아니며, 문서상 확장 방향으로만 정의한다.

### 4.1 2차 확장

- **실제 수집기(collector)**: 여러 출처에서 딜을 자동 수집. 구조는 `docs/05-collector-design.md`.
- **중복 딜 제거(dedup)**: 출처는 다르지만 동일한 딜을 묶음. 초기 스키마의 `source + external_id`, 정규화 제목 해시를 활용.
- **관심 키워드 알림**: 관심 키워드에 매칭되는 신규 딜 발생 시 알림.
- **Redis 도입**: 중복 수집 방지, 캐시, 작업 큐, rate limit 등이 필요한 시점에 추가(`docs/05` 참고).
- **collector worker 분리**: 수집 대상이 늘면 backend API와 collector를 별도 컨테이너로 분리(`docs/02`, `docs/06` 참고).

### 4.2 3차 확장

- **댓글 반응 요약**: 출처의 댓글을 수집·요약해 딜의 평판을 한눈에 제공(AI 요약 구조는 `docs/05`).
- **AI 기반 구매 판단 보조**: 가격 추이, 댓글 요약, 사용자 선호를 종합해 "지금 살 만한가"를 보조 판단.
- **멀티유저 전환**: 인증 도입 및 사용자별 설정 분리(아래 5장에서 미리 대비).

## 5. 사용자 모델 (MVP: 단일 사용자)

- MVP는 **인증이 없는 단일 사용자** 전제로 동작한다.
- 그러나 설정성 테이블(`source_visibility`, `keyword` 등)에는 **`user_id` 컬럼을 미리 두고**, MVP에서는 고정값(예: `1` 또는 `default`)을 사용한다.
- 이렇게 하면 향후 멀티유저로 전환할 때 컬럼 추가 마이그레이션 없이 인증 레이어만 얹으면 된다.
- 상세 스키마는 `docs/04-database-design.md` 참고.

## 6. 비기능 요구사항 (NFR)

- **성능**: 목록 조회는 페이지네이션 기준 1페이지(예: 20건) 응답이 일반 트래픽에서 빠르게 반환되어야 한다(인덱스 설계는 `docs/04`).
- **확장성**: backend와 scheduler를 단일 애플리케이션으로 시작하되, collector worker 분리가 가능한 구조를 유지한다.
- **이식성**: 로컬은 Docker Compose로 DB를 띄우고, 운영은 Vercel(frontend) + OCI/VPS Docker Compose(backend/DB)로 배포 가능해야 한다(`docs/06`).
- **유지보수성**: frontend/backend를 monorepo로 분리하고, 문서 기반으로 구현 순서를 따른다.
- **데이터 정합성**: 동일 딜 중복 저장을 막기 위한 유니크 제약(`source_id + external_id`)을 초기부터 둔다.

## 7. 용어 정의 (Glossary)

| 용어 | 정의 |
| --- | --- |
| 딜(Deal) | 하나의 핫딜 정보. 제목, 가격, 출처, 링크 등을 가진다. |
| 출처(Source) | 딜이 게시된 커뮤니티/사이트. 표시/숨김 설정의 대상. |
| 관심 키워드(Interest keyword) | 해당 키워드를 포함하는 딜만 노출시키는 화이트리스트성 키워드. |
| 제외 키워드(Exclude keyword) | 해당 키워드를 포함하는 딜을 숨기는 블랙리스트성 키워드. |
| dedup 키 | 중복 딜을 식별하기 위한 키(`source_id + external_id`, 정규화 제목 해시 등). |
| collector / worker | 출처에서 딜을 수집하는 (향후) 구성요소. |
