# 0001. 닫힌 집합은 코드 enum으로 — 카테고리 통합 분류 DB 테이블 보류

- 상태: 채택
- 날짜: 2026-06-09

## 맥락
`DealStatus`, `KeywordType` 같은 값 집합과 딜 카테고리를 어떻게 관리할지 정해야 했다.
과거 쇼핑몰 프로젝트에서는 카테고리를 DB 테이블로 운영한 경험이 있어, PickDeal에서도 enum 대신 DB 테이블(룩업)로 갈지 검토했다.

핵심 제약: 출처(커뮤니티)마다 카테고리 체계가 제각각이거나 아예 없다. 이를 PickDeal의 통합 분류로 매핑하는 일이 진짜 비용인데, 그 매핑 규칙을 아직 정할 단계가 아니다(수집기 미도입). MVP에서 카테고리의 주 용도는 화면의 단순 필터다.

## 결정
- **닫힌 집합**(값 추가 = 처리 로직 추가 = 배포인 도메인: `DealStatus`, `KeywordType`)은 **코드 enum + `@Enumerated(STRING)`**으로 관리한다. DB 룩업 테이블을 만들지 않는다.
- **카테고리는 자유 문자열**로 저장하고, `GET /api/v1/deals?category=`는 그 문자열에 대한 단순 옵션 필터로 둔다. **통합 분류 체계·매핑은 수집기 단계로 보류**한다.

## 기각한 대안
- **카테고리/상태를 DB 룩업 테이블로**: 런타임에 값 추가가 가능하지만, 닫힌 집합은 어차피 값마다 코드 분기·배포가 따르므로 테이블의 유연성이 헛이득이다. 카테고리는 "통합 분류"가 전제인데 그 매핑 규칙이 없는 지금 테이블만 만들면 빈 골격이 된다(MVP 범위 초과).
- **카테고리 enum**: 출처별로 분류가 달라 닫힌 집합이 아니다 — enum이 맞지 않는다.

## 결과 / 트레이드오프
- 이득: 스키마·조회 구조가 단순해지고, MVP의 화면 분류 욕구는 키워드/출처 필터 + 단순 category 필터로 충족된다.
- 비용: 통합 분류가 필요해지는 수집기 단계에서 매핑 레이어를 새로 설계해야 한다(그때가 적기). enum 값 추가는 배포를 동반한다.
- 되돌림: 자유 문자열 → 정규화된 분류로 가는 마이그레이션은 나중에 가능하다(지금 테이블을 미리 만드는 것보다 싸다).
