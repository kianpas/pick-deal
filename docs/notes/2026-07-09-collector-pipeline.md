# 수집기 파이프라인 — 작업 기록 & 개념 정리 (2026-07-09)

> 학습/이해용 기록. 확정 설계는 `docs/05`, 진행 규칙은 `AGENTS.md` 참고.
> 출처 사이트 이름은 의도적으로 생략 — 어떤 출처든 동일한 구조가 적용된다.

## 전체 그림

수집기의 목표는 단순하다: **출처 사이트의 게시판 → 우리 DB의 Deal 테이블**.
이걸 한 번에 하지 않고 단계로 쪼갠 것이 파이프라인이다 (`docs/05` A.2):

```
fetch → parse → normalize → persist → (scheduler가 주기 실행)
```

각 단계를 비유하면:

| 단계 | 역할 | 비유 |
| --- | --- | --- |
| fetch | HTTP로 게시판 HTML을 받아온다 | 배달부 |
| parse | HTML에서 제목·가격·링크 등을 구조화된 데이터로 추출 | 번역기 |
| normalize | 출처별 데이터를 우리 표준 `Deal` 형태로 변환 | 통역사 |
| persist | 저장하되 같은 딜을 두 번 넣지 않음 (upsert) | 창고지기 |
| scheduler | 위 과정을 주기적으로 자동 실행 (`@Scheduled`) | 알람시계 |

## 왜 parse를 순수 함수로 만들었나

파서는 `String html → List<딜 항목>` 변환만 한다. **인터넷을 전혀 모른다.**

- 테스트할 때 실제 사이트 접속이 필요 없다. 실제 응답 HTML을 파일(픽스처)로
  저장해두고, 그 파일을 입력으로 넣어 결과만 검증한다.
- 사이트가 느리든 죽어 있든 테스트는 항상 같은 속도로 같은 결과를 낸다.
- 사이트가 HTML 구조를 바꾸면? 새 HTML을 픽스처로 다시 저장하면 테스트가
  즉시 빨간불이 되어 어디가 깨졌는지 알려준다.

픽스처 위치: `backend/src/test/resources/fixtures/`

## 오늘 한 작업 (parse 단계 완성)

TDD(red → green → refactor)로 목록 파서를 구현했다. 검증한 행동 5가지:

1. 목록 HTML에서 **공지를 제외한** 딜 항목 추출 (externalId, 절대 URL 복원)
2. 제목의 `[쇼핑몰]` 접두사를 storeName으로 분리, 제목은 깨끗하게
3. 가격 문자열 파싱 — 콤마, `원` 접미 변형, 무료(0원) 처리. 형식이 어긋나면
   null로 두고 죽지 않음 (수집은 한 건 실패해도 나머지가 살아야 한다)
4. 카테고리·썸네일 URL 추출 (캐시버스터 `?` 제거)
5. 종료 라벨이 붙은 딜을 `ended=true`로 구분

산출물: `collector` 패키지의 `~ListParser`(변환기) + `~DealItem`(record, 파싱 결과).

## 중복 저장은 어떻게 막나 (persist 단계의 핵심)

수집기는 같은 게시판을 주기적으로 다시 읽으므로 매번 대부분이 "이미 본 딜"이다.

- 1차 방어: DB의 `source_id + external_id` **유니크 제약** (스키마에 이미 존재).
  같은 출처의 같은 글번호는 물리적으로 두 번 저장될 수 없다.
- 로직: 이미 있으면 skip 또는 갱신(가격 변동, 종료 전환), 없으면 insert.

이게 MVP 스키마를 설계할 때 dedup 키를 미리 넣어둔 이유다 — 나중에 붙이면
비싸고(기존 중복 데이터 정리 필요), 지금 넣으면 공짜였다.

## 결정: 게시 시각은 근사 변환으로

목록 페이지의 게시 시각은 "30분 전", "5시간 전", "07-08" 같은 **상대/축약
표기**뿐이다. 정확한 시각을 얻으려면 글마다 상세 페이지를 방문해야 하는데,
딜 30개면 HTTP 요청이 1회 → 31회로 늘어난다 (출처에 부담, 차단 위험, 느림).

**선택: 수집 시점 기준 근사 변환** ("30분 전" → `수집시각 - 30분`).

- 근거: 이 서비스에서 게시 시각의 용도는 최신순 정렬이고, 정렬에는 몇 분
  오차가 문제되지 않는다.
- 정확한 시각이 진짜 필요해지는 시점(예: 가격 추이 분석)에 상세 방문을
  붙인다. "나중에 싸게 붙는 것은 미룬다" (AGENTS.md 설계 원칙).

## 같은 날 이어서 완성한 조각 (파이프라인 전체 가동)

- **fetch**: HTTP 클라이언트(`~Client`). 브라우저 User-Agent 필수 — 기본 UA는
  차단됨. 목록 1페이지만 요청한다(출처 부담 최소화).
- **normalize + persist**: `~CollectService`. 파싱 결과 → `Deal` 매핑(통화 KRW,
  종료 딜 → `EXPIRED`, 상대 시각 근사 변환), (source, externalId) 존재 조회로
  신규는 insert / 기존은 가격·상태만 갱신(upsert).
- **scheduler**: `CollectScheduler` — 기동 15초 후 첫 수집, 이후 10분 간격.
  수집 실패는 로그만 남기고 다음 주기에 재시도(스케줄러가 죽지 않게).
  테스트에서는 프로퍼티로 꺼서 실제 HTTP가 나가지 않게 한다.
- **검증**: 테스트에서 HTTP 클라이언트만 목으로 바꾸고 실제 픽스처 HTML로
  "수집 → 저장", "재수집 시 중복 없음", "가격/상태 변경 갱신"을 확인.
  이후 실제 기동으로 라이브 수집 → DB 저장 → 목록 API 노출까지 확인했다.

## 배운 것 / 주의점

- `@SpringBootTest` 테스트들이 같은 in-memory DB를 공유하면 한 테스트의 수집
  결과가 다른 테스트를 오염시킨다 → 테스트 클래스에 `@Transactional`을 붙여
  각 테스트 후 롤백.
- 수집기의 파싱은 "한 건이 이상해도 전체가 죽지 않게" — 가격 형식이 어긋나면
  null, 시각 해석이 안 되면 수집 시각으로 대체.
