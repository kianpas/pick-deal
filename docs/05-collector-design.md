# 05. 수집기 / 확장 방향 (Collector — 의도적 최소 문서)

> ⚠️ 이 문서는 **일부러 짧게** 둔다.
> 수집기·dedup·AI 요약·worker 분리는 전부 2차 이후 작업이고, **1차(MVP)가 끝나기 전에는 상세 설계를 쓰지 않는다.**
> 아직 만들지 않은 것을 완성형으로 설계해두면, 실제 출처를 붙일 때 거의 다시 쓰게 된다. 그래서 "방향"만 적는다.
> 작성 기준일: 2026-05-20

---

## 1. 단계 구분 (요약)

| 항목 | 단계 | MVP |
| --- | --- | --- |
| 실제 수집(crawl/fetch), 정규화 | 2차 | ❌ |
| 교차 출처 중복 제거(dedup) | 2차 | ❌ (스키마만 대비) |
| Redis / worker 분리 / 알림 | 2차 | ❌ |
| AI 댓글 요약 / 구매 판단 보조 | 3차 | ❌ |

MVP는 더미/수동 데이터로만 동작한다(`docs/01`, `docs/04`).

---

## 2. 2차에 들어갈 때의 방향 (한 문단)

단일 Spring Boot 앱 안의 `@Scheduled` 잡으로 시작한다. 출처별 차이는 `SourceFetcher` 인터페이스 구현체로 흡수하고(새 출처 = 새 구현체), 파이프라인은 **fetch → parse → normalize → dedup → persist** 순서로 둔다. 중복은 이미 있는 `source_id + external_id` 유니크로 1차 차단하고, 교차 출처 중복만 정규화 제목 해시(`title_norm_hash`)로 후보를 묶는다. 수집 부하가 조회 API에 영향을 줄 만큼 커지면 그때 worker 컨테이너로 분리하고, 그 시점에 Redis(캐시/큐/중복 방지)를 도입한다. AI 요약·구매 판단은 3차에서 별도 설계한다.

**상세 설계는 1차 완료 후, 실제 출처 1~2곳을 붙여보고 나서 이 문서를 다시 쓴다.**
