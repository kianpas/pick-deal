# Pick Deal 수집기 및 AI 확장 설계 초안

## 1. MVP 범위

MVP에서는 실제 크롤링, 댓글 수집, AI 요약을 구현하지 않는다.

초기 단계에서는 다음 방식으로 데이터를 준비한다.

- DB seed 데이터
- 개발자가 수동 등록한 샘플 데이터
- 추후 관리자 API 또는 간단한 관리 스크립트

이 문서는 후속 확장을 고려한 방향만 정의한다.

## 2. 수집기 목표

향후 수집기는 여러 커뮤니티/사이트에서 핫딜 정보를 가져와 DB에 저장한다.

수집 대상 후보:

- 커뮤니티 핫딜 게시판
- 쇼핑몰 특가 페이지
- 가격 비교 사이트
- 제휴 링크가 없는 원문 링크 기반 게시글

수집 데이터 후보:

- 제목
- 가격
- 배송비
- 원문 URL
- 출처 내 게시글 ID
- 작성/게시 시각
- 품절/종료 여부
- 댓글 목록

## 3. 수집기 아키텍처 후보

### 1단계: 백엔드 내부 스케줄러

```text
Spring Boot API
  -> Scheduler
    -> Source Collector
    -> Database
```

장점:

- 구현이 단순하다.
- 배포 단위가 하나라 운영 부담이 낮다.
- MVP 이후 빠른 실험에 적합하다.

단점:

- API 서버와 수집 작업의 장애 영향 범위가 겹친다.
- 수집량이 늘면 API 응답 성능에 영향을 줄 수 있다.

### 2단계: 별도 Worker 분리

```text
Spring Boot API
  -> Database

Collector Worker
  -> External Sites
  -> Database
```

장점:

- API 서버와 수집 작업을 독립적으로 확장할 수 있다.
- 수집 실패가 API 서버에 직접 영향을 주지 않는다.
- 작업 큐 도입이 쉽다.

단점:

- 배포와 모니터링 복잡도가 올라간다.

## 4. 수집기 인터페이스 초안

출처별 수집기는 동일한 결과 모델을 반환하도록 설계한다.

```text
Collector
  collect(source): List<CollectedDeal>

CollectedDeal
  sourceId
  originalId
  title
  description
  price
  shippingFee
  originalUrl
  postedAt
  status
```

중복 판단 기준:

- 우선순위 1: `source_id + original_id`
- 우선순위 2: `source_id + original_url`
- 우선순위 3: 제목/가격/게시 시각 기반 유사도 비교

MVP 이후 첫 수집기에서는 `source_id + original_url` 기준으로 충분하다.

## 5. 수집 실행 이력

수집 작업은 실행 이력을 남겨야 한다.

상태 후보:

- `RUNNING`
- `SUCCESS`
- `PARTIAL_SUCCESS`
- `FAILED`

기록 항목:

- 출처
- 시작 시각
- 종료 시각
- 신규 수집 건수
- 갱신 건수
- 실패 사유

## 6. 사이트별 수집 전략

### HTML 파싱

- 서버에서 HTML을 요청한다.
- CSS selector로 목록과 상세 데이터를 파싱한다.
- 구조 변경에 취약하므로 selector를 출처별 설정으로 관리한다.

### API 기반 수집

- 사이트가 공개 API 또는 JSON 응답을 제공하는 경우 사용한다.
- HTML 파싱보다 안정적이다.
- 요청 제한과 약관을 확인해야 한다.

### 브라우저 자동화

- 동적 렌더링이 강한 사이트에 사용한다.
- 비용과 복잡도가 높으므로 최후순위로 둔다.

## 7. AI 댓글 요약 확장 방향

AI 요약은 댓글 수집이 안정화된 뒤 추가한다.

처리 흐름:

```text
Collector
  -> deal_comments 저장
  -> Summary Job 생성
  -> AI Summary Worker
  -> deal_comment_summaries 저장
```

요약 결과 후보:

- 한 줄 요약
- 장점
- 단점
- 구매 전 주의사항
- 가격 평가
- 품절/대체 상품 언급 여부

주의사항:

- 댓글 원문은 노이즈와 중복이 많으므로 전처리가 필요하다.
- 개인정보 또는 민감정보가 포함될 수 있으므로 저장 범위를 제한한다.
- AI 결과는 사실이 아닐 수 있으므로 "댓글 기반 요약"임을 UI에서 명확히 표시한다.
- 원문 댓글 일부를 그대로 노출하지 않는 방향을 우선한다.

## 8. 운영 정책

- 출처별 요청 주기를 다르게 설정한다.
- 동일 사이트에 과도한 요청을 보내지 않는다.
- robots.txt, 서비스 약관, 요청 제한을 확인한다.
- 실패한 수집 작업은 지수 백오프 방식으로 재시도한다.
- 수집 장애가 전체 서비스 장애로 번지지 않도록 격리한다.

## 9. 후속 기능 후보

- 품절 자동 감지
- 가격 변동 추적
- 관심 키워드 기반 알림
- 제외 키워드 자동 추천
- 중복 핫딜 병합
- 유사 상품 묶음
- 커뮤니티 반응 점수화
